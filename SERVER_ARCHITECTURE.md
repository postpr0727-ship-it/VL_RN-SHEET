# 서버 아키텍처 설명

## 현재 구조

### 이미 GitHub에 있는 서버 코드

```
api/
├── health.ts              # Firebase 연결 상태 확인 API
├── schedules/
│   ├── index.ts          # 근무표 목록 조회/저장 API
│   └── [id].ts           # 특정 근무표 조회/삭제/수정 API
└── lib/
    ├── firebase.ts       # Firebase 연결 설정
    └── types.ts          # 타입 정의
```

### 배포 방식

- **GitHub**: 코드 저장소 (이미 서버 코드 포함)
- **Vercel**: 자동 배포 및 Serverless Functions 실행
  - GitHub에 푸시하면 자동으로 배포
  - `api/` 폴더의 파일들이 자동으로 API 엔드포인트가 됨

## 작동 방식

1. **코드 작성** → GitHub에 푸시
2. **Vercel 자동 감지** → 서버 코드를 Serverless Functions로 변환
3. **배포 완료** → `https://vl-rn-sheet.vercel.app/api/*` 엔드포인트 사용 가능

## 현재 문제점

Firebase 설정이 필요해서 복잡합니다:
- Firebase 프로젝트 생성
- Firestore 데이터베이스 생성
- 서비스 계정 키 생성
- Vercel 환경 변수 설정
- 재배포

## 대안: Firebase 없이 사용하는 방법

Firebase가 복잡하다면, 다음과 같은 간단한 방법들을 사용할 수 있습니다:

### 옵션 1: 파일 내보내기/가져오기 (가장 간단) ⭐

서버 없이 JSON 파일로 저장하고 불러오는 방식

**장점:**
- 설정 불필요
- 서버 비용 없음
- 파일로 백업 가능

**단점:**
- 브라우저 간 자동 동기화 안 됨
- 파일을 직접 관리해야 함

### 옵션 2: 현재 Firebase 유지

현재 구조를 유지하되, Firebase 설정 가이드를 따라 진행

**장점:**
- 모든 브라우저/기기에서 자동 동기화
- 서버 비용 거의 없음 (무료 티어)

**단점:**
- 초기 설정 필요

### 옵션 3: 다른 서비스 사용

- Supabase (PostgreSQL 기반, 더 간단한 설정)
- PlanetScale (MySQL 기반)
- 등등...

## 추천

Firebase 설정을 완료하는 것을 권장합니다:
- 한 번만 설정하면 계속 사용 가능
- 모든 브라우저/기기에서 자동 동기화
- 무료 티어로 충분함

하지만 설정이 복잡하다고 느끼신다면, **파일 내보내기/가져오기 기능**을 추가하는 것도 좋은 방법입니다!
